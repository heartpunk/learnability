# Correctness Gap Analysis: Bisimulation → Grammar Recovery

## Date: 2026-03-13

## Context

After getting the stratified fixpoint converging (7 functions, 153 blocks, 30ms, correct
counts) and EBNF extraction matching 20/20 golden grammar productions, we analyzed what
exactly is proved vs assumed in the full pipeline.

## What We Have Proved

1. **Bisimilarity**: The extracted LTS has the same observable behavior as the concrete
   binary. Every trace in the LTS corresponds to a real execution trace, and every real
   execution trace is represented in the LTS.

2. **BranchClassesStable**: The fixpoint converges within 2^|closure| iterations.
   Justified by PC-signature finiteness (pigeonhole on finite specification variety).

3. **Soundness**: Every extracted branch corresponds to a realizable execution path
   (`dispatch_bodyBranchSound`).

## The Gap: LTS → EBNF

The EBNF extraction sits on top of the proved LTS. The question: does the extraction
preserve the language?

### Key Insight: The Gap Is Smaller Than It Appears

The EBNF extraction is NOT heuristic pattern matching. Each step is structural:

- **Nonterminal = function.** Read from the binary's call/ret structure, not inferred.
- **Production = converged branch path through function.** These ARE the LTS transitions,
  grouped by function.
- **Terminal = call to tokenizer.** Identified by call target address (structural).
- **Terminal label = guard condition at call site.** From symbolic PC (exact).
- **Character class = tokenizer DFA branch condition.** Derived from tokenizer's own
  branches, NOT from external knowledge.

### No Binary-Specific Knowledge Is Injected

Everything is derived from the binary's own structure:

- Tokenizer identified structurally (leaf function that reads input byte-by-byte, called
  by all NT functions).
- Integer-to-token mapping derived from tokenizer's own branches (e.g., when input is in
  `[a-z]`, tokenizer returns 14 in rax → token 14 = the `[a-z]` character class).
- Parser checks `CmpEQ32(rax, 14)` → structural connection to tokenizer's `[a-z]` path.
- Human-readable labels ("id", "int") are cosmetic. The `GrammarSym` type carries
  `CharClass`, not strings. Golden comparison uses structural isomorphism, not labels.

The only architecture-specific knowledge is the calling convention (rax = return value on
x86-64), which falls under the ISA trust boundary.

## Provable Properties

### Property 1: Conservative Parser Detection (DO FIRST)

**Statement:** Define sufficient conditions for "this binary is a parser":
- Has a function that reads input byte-by-byte and returns an integer (tokenizer)
- Has other functions that call the tokenizer and each other (NT functions)
- NT functions branch on integer comparisons of the tokenizer's return value
- Call/return structure follows the architecture's calling convention

**Guarantee:** Whenever the check says "yes, this is a parser," it is always correct.
It may miss exotic parsers (false negatives OK), but never false-positives.

This is a conservative (sufficient-condition) definition. A binary that passes this
check definitely has the structure our extraction assumes.

### Property 2: EBNF = Language(LTS)

**Statement:** The language generated by the extracted EBNF equals the language accepted
by the LTS.

**Proof approach:** Structural argument. The EBNF productions correspond 1-1 to LTS path
classes (converged branches grouped by function). No fixpoints or coinduction needed —
finite structural induction on the extraction procedure.

**Combined with bisimilarity:** EBNF generates exactly the language of the concrete parser.

### Property 3: ISA-Precise Trust Boundary (FUTURE)

**Current state:** Trust boundary = pyvex correctly lifts x86-64 to VEX IR.

**Future:** ISA-precise symbolic execution (cf. Park 2025, filtered simulation for binary
lifting). Trust boundary narrows to: "the ISA specification is correct."

With a trusted ISA spec:
- "Function" = code reachable from a call instruction that returns via ret (structural
  property of ISA semantics)
- Standard calling conventions become provable properties of the ISA
- Can prove: "we detect all functions in standard calling styles for a given ISA,
  presuming a valid spec thereof"

Non-standard call management (e.g., hand-rolled calling conventions, coroutines) would
require manual annotation. This is acceptable — the tool refuses or asks for help rather
than silently producing wrong results.

## The Full Provable Chain

1. Conservative parser detection (sufficient conditions) — **provable, do first**
2. LTS extraction via bisimilar fixpoint — **proved**
3. EBNF = Language(LTS) — **provable** (structural argument)
4. Therefore: EBNF = Language(concrete parser) — **follows from 1-3**

Trust boundary: ISA semantics (pyvex today, ISA-precise later). Everything above that
boundary is derivable and provable.

## Relationship to ML Concerns

Sophie's concern: "doesn't trying more targets before proving everything risk training on
the test set?"

**Answer: No.** This is not ML. The proofs hold for ALL binaries, not just the ones we
tested on. Running on more targets:
- Stress-tests the engineering (missing opcodes, edge cases)
- Cannot "overfit" the proofs (they're universal)
- Reveals what actually needs to be different vs what we imagine

The right order: state Property 1 (conservative parser detection) precisely, then try more
targets. Property 2 (EBNF = Language(LTS)) can be proved in parallel. Properties are
conjectures until proved, but they're mathematical conjectures tested by examples, not
statistical claims vulnerable to overfitting.

## WTO Architecture Status (for reference)

Block-level WTO was attempted (2026-03-13) as an optimization for ordering within the
fixpoint. Results:

- **Multi-function SCC architecture works** for small cases (13 blocks, 5 functions,
  4 outer rounds, correct convergence).
- **Fails for deeply nested single-function SCCs** (32-block statement SCC, 8 levels of
  nesting, 2254 branches at K=0 vs 214 expected, OOM at 10GB).
- **Root cause:** Block-level composition within a single function produces combinatorial
  blowup without abstraction boundaries. Function-level approach already HAS the right
  abstraction (function summaries at call boundaries).
- **Decision:** Deferred. Function-level approach (30ms, correct counts) used for all
  targets. Block-level WTO is a performance optimization for very large binaries — worth
  revisiting when we have concrete evidence it's needed.

The valuable infrastructure (return edges, WTO computation, CFG building) is preserved
in the learnability-soundness workspace for future use.
